# Git 提交规则

本流程适用于非管理员对仓库文件的所有变更。规则文件只是重点审核对象，因为错误规则会影响所有 agent 的行为。

## 核心原则

`main` 是正式版本。非管理员不能直接把改动合进 `main`，需要先发起 PR，由管理员审核后再合并。

PR 可以理解为“变更申请”。PR 合并前，改动只是候选变更；PR 合并后，才进入正式版本。

## 开发人员怎么做

1. 从最新代码创建自己的分支。
2. 修改需要变更的文件。
3. 在自己的分支正常提交，可以提交多次。
4. 推送自己的分支。
5. 创建 PR。
6. 在 PR 里说明改了什么、为什么改。
7. 如果改了 `AGENTS.md` 或 `rules/**`，额外说明是否可能和已有规则冲突。

开发人员不需要每次提交都找管理员确认，也不需要每次提交都手动执行一堆检查命令。

## 管理员怎么做

1. 查看 PR 里的改动。
2. 对普通文件按正常代码或文档质量审核。
3. 对 `AGENTS.md` 和 `rules/**`，重点判断规则是否有效、是否重复、是否和已有规则冲突。
4. 必要时要求开发人员调整。
5. 审核通过后合并 PR。

## 一句话版本

其他人可以在自己的分支随便改、随便提交；只要改动要进入 `main`，就必须通过 PR 让管理员审核后合并。
