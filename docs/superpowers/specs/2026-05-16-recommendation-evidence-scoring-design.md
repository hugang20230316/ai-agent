# 推荐证据评分规则设计

## 背景

全局 `skill-rules.md` 里的推荐规则原本把“官方维护状态”写进采用度信号，又把用户随口举的 `1000` 写成主推荐硬门槛。这个规则会产生两个问题：

- 没查使用量时，也可能因为“官方”或“描述看起来合适”被写成推荐。
- 绝对数字会误伤小生态，也会放过大生态里的弱候选。

本设计把推荐逻辑改成分层证据评分。目标不是追求一个死阈值，而是让每次推荐都能说明：为什么它适合当前需求，证据够不够，风险在哪里。

## 目标

- 推荐前必须先收集证据，再给结论。
- 主推荐必须有采用度证据，但不使用固定安装量门槛。
- 官方、可信来源和活跃维护只能加分，不能替代采用度。
- 输出必须区分主推荐、候选和不推荐。
- 缺证据时要明确说缺什么，不能把候选包装成推荐。

## 非目标

- 不要求每次都做完整打分表。简单场景可以给简化版，但证据维度不能缺。
- 不要求所有工具都有公开安装量。没有公开数据时，可以用下载量、Star、生态排名、官方目录收录等替代证据。
- 不自动安装任何 skill、插件、库或工具。

## 评分模型

每个候选按五个维度判断：

| 维度 | 权重 | 证据 |
| --- | ---: | --- |
| 采用度 | 35 | 安装量、下载量、Star、生态使用量、同类排名 |
| 来源可信度 | 20 | 官方、头部组织、知名维护者、明确归属 |
| 维护状态 | 15 | 最近更新、版本发布、issue/PR 活跃、是否废弃 |
| 功能匹配 | 20 | 是否解决当前需求，是否与已有能力重复 |
| 风险 | 10 | 权限、供应链风险、侵入性、是否会让 agent 更啰嗦或误触发 |

采用度采用相对判断：先看候选所在生态和同类竞品，再判断它是否足够主流。不同生态不能共用一个固定安装量门槛。

## 输出分类

| 分类 | 条件 | 输出要求 |
| --- | --- | --- |
| 主推荐 | 采用度证据充分，综合证据强，风险可接受 | 写名称、用途、采用度证据、来源和推荐理由 |
| 候选 | 功能匹配但采用度不足、查不到采用度，或风险还需确认 | 说明缺少哪类证据，不能写成“建议直接装” |
| 不推荐 | 采用度弱、来源弱、长期不维护、风险高，或和已有能力重复 | 说明淘汰原因 |

## 推荐流程

1. 明确用户要解决的问题和使用场景。
2. 搜索候选，并优先选择同类主流方案。
3. 对每个候选收集采用度、来源、维护、匹配度和风险证据。
4. 用同类相对排名判断采用度，不使用固定数字。
5. 分成主推荐、候选、不推荐。
6. 如果没有主推荐，先说明已有能力或自建规则方案，不默认安装弱候选。

## 示例口径

推荐 React skill 时，几十万安装量的头部 skill 可以主推，几百安装量的同类 skill 即使功能描述匹配，也只能列为候选或不推荐。

推荐冷门 MCP skill 时，如果整个生态公开安装量都低，可以参考官方目录、GitHub Star、最近更新和同类排名。采用度弱但来源强的项目，最多列为候选，除非没有更成熟替代，并且要说明这是小生态里的相对选择。

## 规则改动

把 `skill-rules.md` 的“推荐与筛选”改为：

- 推荐前必须先收集证据，不允许只凭名称、官方来源或描述推荐。
- 主推荐必须有采用度证据，但采用度按同类相对水平判断，不设置固定数字门槛。
- 采用度、来源可信度、维护状态、功能匹配和风险都要进入判断。
- 官方、可信来源、活跃维护不能替代采用度，只能作为加分项。
- 输出必须区分主推荐、候选和不推荐。
- 没有主流候选时，优先说明已有能力或自建方案。

## 验收标准

- `skill-rules.md` 不再把任何固定数字写成主推荐门槛。
- 规则明确要求“先查证据，再推荐”。
- 规则明确采用度是相对判断，不同生态不能共用固定数字。
- 规则明确官方身份不能替代采用度。
- 规则明确输出分为主推荐、候选、不推荐。
