# Agent 项目治理规则

- 公共同步仓库只保存通用规则、说明和无敏感信息的 memory。
- 个人规则仓库中只有 `rules/*.md` 制定公共或工具行为规则；`docs/*.md` 只保存同步说明、迁移说明和文件分类，不参与规则加载。
- 规则分为四层：公共规则放 `rules/*.md` 并默认加载；工具规则放 `rules/<tool>-rules.md` 并只在当前运行工具或工具相关任务中加载；项目规则放目标项目的 `AGENTS.md` 和 `.codex/rules/`；本机私有配置放 `local/`、环境变量或各工具 profile。
- 共享 Agent 仓库的 `AGENTS.md`、`rules/` 和 `skills/` 必须以仓库自身位置为源；工具目录只作为安装或链接目标，不能作为公共源路径。
- 工具如何接入公共规则可以写在公开接入说明或本机 profile；公共规则正文不写机器路径、工具本地目录、账号、私有启动命令或个人目录结构。
- 个人自建或明确个人维护的 skill 归入 `skills/<skill-name>/`，工具侧只通过软链接或配置引用加载。
- 普通任务、规则纠偏、配置调整、测试检查、排查修复和一次性决策不自动写设计文档、spec、plan、报告、复盘文档或留档文件；除非用户明确要求留档，或任务本身就是正式方案/需求/架构文档。
- 用户明确要求使用某个执行型 skill 时，该 skill 内部要求的计划拆分、review 或 final review 属于执行流程和质量门禁，不等同于写入项目的 spec、plan、报告或留档文件；除非用户要求留档，否则只在会话内执行和汇报结果。
- 用户目标是“发现问题、解决问题”时，优先输出问题、证据和修复动作；不要把中间流程文档当成交付物。
- 项目规则只放目标项目仓库的 `AGENTS.md` 和 `.codex/rules/`。
- 任何规则修改都必须走 `rule-fix`：写入前展示拟改文件、修改理由和排除项，写入后做规则归属、重复、冲突、硬编码和隔离验证。
- 当前会话中用户明确指出规则问题并授权继续处理时，走即时规则热修：先复述失败模式，再查现有规则、确定归属、修改或合并规则并验证；不需要先进入 Obsidian 候选确认。
- 规则热修前先做覆盖诊断：列出已覆盖该失败模式的现有规则、说明失效点属于触发、加载、冲突、验证还是规则缺口。已有规则已覆盖但未生效时，优先修触发、加载或验证方式，并改原规则的触发和执行门槛；只有现有规则没有覆盖可复用行为时，才新增规则条目。
- 规则要短，直指失败模式和执行动作；能用一句话说清就不要写成长段。
- 写规则前先查同文件是否已有类似规则；有类似规则时先判断为什么没生效，优先改原规则，不直接追加。
- 规则只写可复用原则，禁止写硬编码业务名、字段名、接口名、人员名、工具特例或一次性场景。
- 规则候选不得依赖具体话术、业务名或事故细节；依赖这些才能生效的候选必须退回。
- 历史扫描、模糊模式和未确认偏好只进 Obsidian 候选；Obsidian 候选、Daily、历史会话和原始日志只能作为规则修改证据，不能自动改变规则。
- 从 Obsidian 候选升级规则前，先核对状态、验证情况、关联问题、疑似模式和人工确认；符合稳定、可复用、会影响 agent 行为这三项时才改 `rules/` 或自定义 skill。
- 规则写完前检查：无重复、无冲突、能被同义请求和反向请求触发。
- 规则写完后必须按 `rule-fix` 用空上下文子 agent 做多场景验证；没有完成验证前，不能声称规则已生效。
- 若同一任务同时涉及全局规则和项目规则，先说明分层方案，再分别落到个人规则仓库和目标项目仓库；没有明确授权时只改个人规则仓库。
- 私有配置、敏感信息、公司项目边界和上传前敏感扫描遵循安全、隐私与同步边界规则。
