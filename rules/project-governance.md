# Agent 项目治理规则

- 公共同步仓库只保存通用规则、说明和无敏感信息的 memory。
- 个人全局规则的实际生效位置是 `~/.codex/rules/*.md`；公开同步仓库只保存这些规则的可迁移副本。
- 个人规则仓库中只有 `rules/*.md` 制定规则；`docs/*.md` 只保存同步说明、迁移说明和文件分类，不参与 Codex 规则加载。
- 个人自建或明确个人维护的 skill 归入 `skills/<skill-name>/`，工具侧只通过软链接或配置引用加载。
- 普通任务、规则纠偏、配置调整、测试检查、排查修复和一次性决策不自动写设计文档、spec、plan、报告、复盘文档或留档文件；除非用户明确要求留档，或任务本身就是正式方案/需求/架构文档。
- 用户明确要求使用某个执行型 skill 时，该 skill 内部要求的计划拆分、review 或 final review 属于执行流程和质量门禁，不等同于写入项目的 spec、plan、报告或留档文件；除非用户要求留档，否则只在会话内执行和汇报结果。
- 用户目标是“发现问题、解决问题”时，优先输出问题、证据和修复动作；不要把中间流程文档当成交付物。
- 项目规则只放目标项目仓库的 `AGENTS.md` 和 `.codex/rules/`。
- 规则要短、清楚、可执行；能用一句话说清就不要写成长段。
- 写规则前先查同文件是否已有类似规则；有类似规则时先判断为什么没生效，优先改原规则，不直接追加。
- 规则只写可复用原则，禁止写硬编码业务名、字段名、接口名、人员名、工具特例或一次性场景。
- 规则写完前检查：无重复、无冲突、能被同义请求和反向请求触发。
- 若同一任务同时涉及全局规则和项目规则，先说明分层方案，再分别落到个人规则仓库和目标项目仓库；没有明确授权时只改个人规则仓库。
- 私有配置、敏感信息、公司项目边界和上传前敏感扫描遵循安全、隐私与同步边界规则。
