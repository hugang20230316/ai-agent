# Agent MCP 输出规则

- 面向用户直接回传 MCP 查询结果时，必须说明查询目标、查询条件、数据源、返回数量和结论。
- 运行 `rg`、`grep`、`find`、日志查询或大范围 API/MCP 查询前，必须先限定目录、文件类型、关键业务词和返回上限；禁止用 `token`、`Header`、`Guid`、`Exception` 这类泛词扫用户 home、整仓或多服务目录。
- 搜索词包含反引号、`$`、引号、管道符或其他 shell 元字符时，必须用固定字符串匹配或安全引用，避免 shell 把搜索词当命令执行。
- 工具输出超过当前问题所需范围时，立即停止沿用该结果，改用更窄查询、`--max-count`、`--files`、`-g` 过滤、`sed` 定位行号或数据库 `LIMIT`；不要继续扩大搜索，也不要把长输出塞回对话上下文。
- 用户查询账号、ID、手机号、编码等是为了继续查数据或调用接口时，结果要优先用控制台友好的短表直接给可复制的原值；不要只给打码值或只给 CSV/附件路径。
- 用户点名 MCP 或具体数据源时，先尝试对应 MCP；失败时说明 MCP 查询目标、失败原因和采用的只读兜底方式。
- 用户要求追接口链、查源头或判断上下游责任时，MCP/日志/API 证据要按数据产生时间线输出：数据源、查询条件、关键入参、关键出参、该层结论。不能只给最终判断，也不能只停在当前查询语句或当前服务层。
- 涉及接口请求、接口排查、接口验证、联调参数或请求/响应比对时，默认按正常请求路径输出：请求地址或路由路径、请求方法、完整入参、完整返回、实际状态和证据来源；凭据类字段按脱敏规则处理，不能只给关键字段、片段或口头结论。
- 用户要求“完整传值、完整入参、完整 payload、原始请求体”时，返回可复现的完整请求对象和完整响应对象；完整指结构、业务定位字段、可复现字段和实际返回结构完整，凭据类字段仍按脱敏规则处理。若日志、API 或 MCP 只记录调用链、截断字段，或没有完整请求/响应体，必须明确说明未记录完整请求/响应体，并列出已查数据源、表/集合、查询条件和缺口。
- 内部排查使用的中间查询不需要套完整模板，但最终结论要说明依据来自哪个数据源。
- 根因结论必须标明证据来源类别，例如代码配置、消息队列拓扑、数据库状态或日志记录；不能只给没有证据的推测。
- 查询输出和敏感业务标识展示遵循安全、隐私与同步边界规则。
- 结果数量默认只展示少量关键行；大结果集用汇总统计说明。
- 若数据源不可用，说明失败原因和已尝试的替代方案，不伪造结果。
- MCP 不可用时，可以使用等价只读 API 或 CLI 兜底；最终答复必须明确这是兜底查询，不是 MCP 直接返回。
- MCP 连接信息以本机配置或本轮实际返回为准；不要在公共规则中写死地址、端口或账号。
