# Agent 编码规则

- 先遵循当前仓库和同文件已有风格、命名、目录结构和测试方式；新增字段、默认值、对象赋值和注释要贴着既有结构写，无明确业务理由不另起写法。
- 只改完成当前目标所必需的代码；发现旁路的安全、日志、重构、格式化或清理问题，先说明建议，不混进当前补丁。
- 遇到本轮开始前已有的改动，默认视为用户或其他流程的改动；不要回滚、覆盖或格式化它们，除非用户明确要求。若这些改动影响当前修复，先兼容处理并说明边界。
- 写代码前必须能从用户最新指令或后续确认中直接说清：允许修改的对象、目标变化和边界；如果只能说出待验证现象、风险或猜测，停在证据和建议，不改代码。
- 测试或验证任务不自动授权任何仓库写入或代码结构调整；只有用户明确把交付物定义为代码修改，或已授权修复且写入属于修复范围时，才允许改文件。
- 修一段逻辑时，必须同步核对同一判断链里的命名、注释、分支条件和辅助方法；这些必要关联修改不算顺手扩大范围。
- 修改功能点前，先按业务语义界定影响范围；沿已有调用、数据流或同类规则确认实际存在的写入、读取、展示和后续消费路径，并说明改、不改或无法确认的原因。
- 修改代码、DTO、接口契约或字段注释前，先追溯同一业务字段在模型、DTO、枚举、映射、读写入口和已有文档中的完整定义；不得只按用户口头示例或页面临时文案改窄语义。
- 迁移接口、方法、DTO 或同类成组契约时，必须把入口、实现、DTO、注释、引用和路由/命名空间作为同一迁移单元检查；用户只点名入口不代表 DTO 或契约可以留在旧归属里。
- 用户要求迁移、移动或调整归属时，只表示改变所属位置和引用关系；没有明确说删除、废弃或清理旧类型/文件/层级时，不得删除原容器或无关同级对象。
- 涉及状态、持久化、查询语义或展示规则时，检查同一数据或状态的读侧、异步处理和反向流程；未找到实际路径时只记录风险，不新增假设性规则。
- 修复完成前，从用户入口到持久化、再到用户可见读取做一次回查；无法验证的相关路径在结论里说明。
- 用户明确要求只改注释、文案或说明时，只能改对应文本；发现校验、查询、赋值、DTO 特性等行为问题，先说明，等确认后再改。
- 不为未来可能需要的功能提前加抽象、接口、配置项或兼容分支。
- 补能力或修缺口前先查现有入口、脚本、配置、接口和同类实现；能复用或修正已有能力时，不重建一套平行实现。
- 性能优化必须先说明瓶颈、基线和对照口径；没有可复现的慢点、测量数据或明确调用成本时，不把重写、并发化、缓存或异步包装当作优化结论。
- `Task.Run`、并发、缓存、批处理或异步包装这类实现改动必须能解释真实收益和风险；如果原方法已经异步或瓶颈不在当前线程，先给证据和建议，不为形式上的“优化”改代码。
- 不要为只使用一次的成员访问、表达式或对象新增临时变量、包装对象或辅助方法；只有复用、避免重复求值或明显提升当前代码可读性时才提取。
- 单次引用且逻辑很短的表达式不要抽成私有方法；私有方法必须服务复用、隔离明确业务步骤，或让当前方法明显少掉一段需要解释的业务流程；否则完成前按调用次数、方法体长度和主方法减少量撤回无价值封装，并按最终 diff 统计新增私有方法是否仍有保留价值。
- 命名要符合业务，简短清楚；Controller 或同类业务入口新增/改名方法前，先查当前类和相邻同类文件的命名证据，没有同类依据时不要把查询键、底层字段、来源表或实现细节塞进方法名。
- 新增接口前必须先判断是否只是现有接口的薄包装；如果只是透传或简单赋值，默认复用现有接口，只有需要隐藏/收窄调用契约或增加明确业务约束时才新增入口。
- 组合查询条件必须先写清楚 AND/OR 关系再实现；筛选项默认是 AND，关键字多字段匹配这类内部候选才使用 OR，并用注释说明。
- 修改方法名、职责或语义后，必须同步检查局部变量和注释命名，避免保留旧语义。
- 注释只写必要信息，短句说明核心含义、业务约束或不明显原因；不要把代码翻译、泛词占位或长段流程说明当成业务定义。
- 写变量名和注释前，先把字段值、枚举值、SQL 条件、接口参数或页面临时叫法翻译成业务语义；不能把实现信号直接写成业务注释。
- 注释必须说明当前值的业务含义、参与判断的原因，或它和相邻分支的关系；如果还说不清，先继续读代码和证据，不要用泛词占位。
- 新增或改写较长局部变量、复杂赋值、关键分支条件时，上一行要有简短注释说明业务含义或判断原因；同一类新增变量和判断要一次性扫全，不要等用户逐个指出。
- 修改字段或 DTO 注释时，必须保留字段基础语义；如需补充使用场景，用括号追加，不要用场景描述或单个页面示例替代字段含义。
- 新增或修改方法时，按当前项目和文件既有文档风格补齐必要注释；对外接口方法注释只写业务定义，参数、返回结构和实现细节交给 DTO 注释或代码本身。
- 新增或修改字段、常量和配置项时，必须有简单明了的注释，说明用途。
- 用户纠正业务术语后，所有本次触碰到的变量名、方法名和注释必须统一使用新术语，并回到相关判断链重新确认业务语义，不能只替换字面词。
- 在 Controller、Service、Business 等类中新增私有辅助方法时，默认放在类底部已有 private helper 区域；除非当前文件已经按功能分区组织，并且新增方法紧邻调用点更符合现有风格。
- 新增或修改代码注释时，优先保持当前文件的标点风格；若当前文件没有固定风格，短注释末尾不加中文句号。
- 完成代码修改后必须检查 `git diff --stat` 和相关文件 diff，确认没有格式化、空白、注释、命名、相邻文件顺手改动、一次性私有方法、额外构造依赖、新接口、新状态组件或提前抽象；发现无关改动、过度封装或过度设计要先撤回再验证。
- 环境地址、账号、密码、token、机器路径、端口和一次性命令的处理遵循安全、隐私与同步边界规则。
- 修改非 UTF-8 或编码不明文件时，先确认编码再改，避免整文件重写造成乱码。
- 对外 API、DTO、契约、数据库结构和共享工具改动要优先考虑兼容性和调用方影响。
