# Agent Markdown 写作规则

## 强制执行流程

- 只要任务涉及创建、修改、审阅或生成 Markdown 文档，必须在动笔或编辑文件前读取本文件；不能只依赖记忆中的规则。
- 读取本文件后，先判断文档是否需要图表：凡是涉及方案、架构、流程、模块关系、部署、原理、对比或决策说明，默认需要图表；除非用户明确要求纯文本。
- 写作前先确定文档要回答的核心问题，再决定图表数量和类型。复杂方案至少使用“背景/问题图 + 原理/流程图 + 落地/结构图”中的两类，不要只放一张粗略结构图。
- 编写前端或 API 对接文档时，默认只写当前已实现且调用方可直接使用的接口、字段、状态和处理流程；若用户明确要求梳理确定需要的缺失接口或完善点，可以单独成节记录，但不得包装成现有可调用契约，也不得扩展未确认方案。
- 前端对接文档必须面向调用方组织内容：正文只写页面或操作位置、接口路径或方法名、请求字段、响应业务字段、状态值、成功后的 UI 处理、刷新或轮询触发条件，以及会阻塞前端实现、联调或验收的具体待确认项。后端统一响应包装、Controller/DTO/Service 归属、实现过程、内部证据和框架说明默认不进入正文；调用方必须解析或判断的响应包装字段、错误码、分页结构除外，应按前端处理口径写入。内部采集过程、代码锚点清单、文件路径清单、agent 自我纠错说明和给作者自己的提醒不得进入正文。
- 写入或修改 API 对接文档前，必须对文档涉及的每一个接口名（控制器/Action 或路由路径）在仓库代码中做一次证据核验：用 grep 或同等检索在 `.cs`、`.ts`、`.tsx`、`.vue`、`.java`、`.kt`、`.go`、`.py` 等实现文件中找到接口的真实定义或路由声明；找不到证据的接口名一律不得进入对接文档主体。一句"当前未实现"的免责说明不能替代证据核验，也不能为继续在主体中描述该接口开口子。
- 前端对接文档的接口表必须区分“当前页面已确认调用”和“接口存在但调用点待确认”。只有前者可以进入当前页面主体流程；后者只能放入边界、待确认或能力说明。
- 写入或修改前端/API 对接流程图前，必须先建立证据矩阵，逐条标明主体里的边属于用户动作、页面跳转、接口调用、局部刷新、状态轮询、异步回流、状态转移、数据依赖还是待确认，并记录对应证据类型。没有证据的边不得进入主体图。
- 前端或 API 对接文档中的流程图必须先声明图表类型：用户流程、接口时序、状态流转、能力关系或数据依赖。不同类型不能混画；用户流程图只允许放已确认的页面动作、页面结果、系统反馈和状态变化。
- 前端或 API 对接文档中的接口时序图、能力关系图、数据依赖图，接口调用节点默认必须统一使用 `中文接口说明（接口地址）`；若用户或既有文档约定了更具体格式，按该格式执行并保持全文一致。没有明确要求时，不得用 Controller/Action 英文方法名代替中文接口说明。用户流程图不得为了展示接口名混入纯接口节点。
- 接口名通过代码证据核验，只能证明接口存在；不能据此把接口调用、数据查询、局部刷新或状态轮询画成页面下一步。保存、提交、完成类动作后的页面回流必须有原型连线、前端路由/组件证据或当前口径；没有证据时只能写待确认，不能进入主体流程图。
- 待确认流程、候选接口和未确认落点不得进入对接文档主体流程图；其他场景能力或兄弟模块流程不得混入当前用户流程或当前场景主线，确需表达时必须独立成能力关系图或数据依赖图，并标清边界。
- 对接文档"主体"明确包含：接口归属表、Mermaid/PlantUML 时序图、场景流程表、入参/出参字段表、请求/返回示例 JSON、状态码表。这些位置只能出现已通过证据核验的接口；未实现接口不得以任何形式出现在主体中，无论是否标注"建议"、"草案"、"未实现"。
- 未实现或仅为讨论建议的接口名，唯一允许出现的位置是文档末尾独立的"缺口/待确认"章节，且必须满足：章节开头用显式警示标明"代码未实现，不可作为对接契约、不可作为联调依据"；只描述缺口现状、待确认事项、影响范围；不得包含路径、方法、入参、出参、状态码、字段表或示例 JSON 等任何契约级细节。
- 同一接口名在文档中允许出现的位置最多两类：实现已存在时进入主体；不存在时仅在末尾缺口章节作为讨论备忘出现一次。不允许同时出现在归属表与缺口章节，也不允许在主体里以"建议"形式预占位。
- 用户未指定其他目录时，长期技术教程默认放到工作区 `docs/tutorials/<知识名称>/README.md`，文件夹按知识名称命名；只有用户明确要求写入 Obsidian 知识库目录，才放到 `docs/obsidian/`。
- 创建或更新教程后，最终答复必须给出教程主体文件的实际落盘路径；不要做成 Markdown 链接，不要加 `:行号`，不要只给相对路径、文件夹路径或审查文件。
- 写完后必须做 Markdown 自检：代码围栏成对闭合、Mermaid 使用保守语法、图表可独立阅读、图表数量覆盖关键概念、正文没有依赖聊天上下文。
- API 对接文档自检还必须额外做一次接口名证据核验：列出文档主体中出现的全部接口名，对每个接口在仓库代码中再 grep 一次确认存在；任一接口找不到证据时，先把它从主体清出去再交付，不得以"已加未实现说明"为由通过自检。
- API 对接文档自检还必须按图表类型检查主体图的每条边是否有对应证据，并确认图表类型没有混用。用户流程图中缺少页面动作、页面结果、原型连线、前端代码或当前口径证据的流程边，必须移出主体图，并放入缺口、边界或待确认章节。
- 前端对接文档自检还必须检查读者边界：正文不能出现后端实现归属、内部代码锚点清单、文件路径清单、采集过程复盘、纠错说明或提醒 agent 自己避免错误的文字；这些内容只能留在会话说明、内部证据或独立复盘中。待确认项必须具体说明缺失决策或证据、影响的前端实现或验收点、确认后会改变什么；缺任一项就删除或移出正文。用户要求图中展示接口/方法名时，还要检查同一类接口调用节点命名是否统一，且已确认的前置查询、候选列表、保存后刷新等接口没有漏掉。
- 最终答复默认不重复说明内部 Markdown 自检。只有复杂 Markdown 文档、含图表/代码围栏/渲染兼容风险的交付、用户要求说明验证过程，或自检发现异常、跳过项时，才说明检查结果和原因。

- 写 Markdown 文档时，优先保证常见预览器可直接渲染；图表、表格和代码块必须能独立阅读，不依赖聊天上下文。
- Markdown 代码围栏必须成对闭合；修改后要检查围栏数量和语言标记是否正确。

## 图表风格

- 能用图清楚表达结构、流程、关系、分类或对比时，优先用图；少写解释性长段落，只保留图前后的必要结论、路径和注意事项。
- 在聊天回复中，不要直接输出 Mermaid、PlantUML、Graphviz、draw.io XML 或其他需要 Markdown/专用渲染器才能识别的图表源码。除非已经确认当前界面能正常渲染，或用户明确要求源码，否则使用文字、表格、编号流程说明，或生成可直接打开的图片/文档文件。
- 如果图表无法在用户当前查看环境中渲染，图表源码对用户没有实际价值；不要把这类源码当作最终说明交付。
- 生成 Markdown 图表时，不要默认使用单调的 Mermaid 原生小方块样式。除非用户明确要求极简源码图，否则优先选择更清晰、更生动的图表风格。
- 面向用户交付的教程如果用户查看环境不能稳定渲染 Mermaid，不能只放 Mermaid 源码；应生成 SVG/PNG 等可直接显示的图片图表并在 Markdown 中嵌入。
- 能用图表清楚表达的教程内容，优先用图表承载主体信息，正文只补充关键操作、边界和验证方法；不要用大段文字重复图中已经表达的关系、流程或分类。
- 对外说明、概念解释、复杂流程总览优先使用 Excalidraw 手绘说明风。
- 正式架构、部署、模块边界和评审材料优先使用 draw.io/diagrams.net 正式架构风。
- 日常技术文档可使用增强配色 Mermaid：使用分组、颜色、节点形状和短标签提升可读性，但保持语法保守，确保 Markdown 预览器能渲染。
- 流程说明优先使用泳道图或时序图风格；如果一个流程图超过 10 个节点、连续长节点超过 5 个，或渲染后需要缩放到一屏内导致文字不可读，必须改成纵向、分组或拆成“总览图 + 关键流程图 + 文件/产物关系图”。
- 复杂图不要强行塞进一个 Mermaid 代码块。适合展示的图生成 SVG/PNG 或 `.excalidraw`/`.drawio` 源文件，再在 Markdown 中嵌入图片并保留源文件路径说明。

## Mermaid 兼容性

- 写 Mermaid 图时使用保守语法：节点 ID 使用简单英文或数字，节点文案使用短文本，连线使用 `-->`、`-->|文本|` 这类基础写法；若节点文案必须包含接口地址、请求方法、`/`、`?`、括号等符号，必须使用 `A["中文接口说明\n（接口地址）"]` 这类带引号的节点标签。
- Mermaid 节点内不要放复杂文件路径、尖括号占位符、引号嵌套、反斜杠换行、HTML 标签、过长中文句子或带说明文字的虚线箭头；这些内容放到正文、表格或图下说明里。
- 需要表达路径、命令、占位符、版本号或长说明时，图里只放概念名，图外用表格列出精确内容。
- 为 Markdown 文档新增或修改 Mermaid 图后，至少做一次静态检查：确认 Mermaid 代码块内没有 `<占位符>`、未转义引号、复杂箭头和未闭合围栏；如果本机有 Mermaid 渲染器，再做渲染验证。
