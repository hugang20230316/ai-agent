# Agent Skill 使用规则

## 触发与加载

- 用户用 `$SkillName`、明文 skill 名称、插件名，或任务语义明显命中某个 skill 描述时，必须按当前会话可用 skill 列表匹配对应 skill。
- 用户要求 AI 操作、排查、验证或管理定时任务、计划任务、LaunchAgent、LaunchDaemon、launchctl、cron、crontab、at、Windows Task Scheduler、PowerShell ScheduledTasks、schtasks 或同类后台定时作业时，默认触发 `timer` skill；除非用户明确要求绕过该 skill，或 `timer` skill 明确不支持目标后端，否则不得直接把底层系统命令作为首选入口。
- 命中 skill 后，先读取对应 `SKILL.md`，只读取完成任务所需的说明；如果 `SKILL.md` 引用脚本、模板或相对路径，按 skill 文件所在目录解析。
- 如果多个 skill 同时命中，选择覆盖任务所需的最小集合，并说明使用顺序。
- 如果用户点名的 skill 不存在、路径不可读或无法应用，必须说明原因，再使用最接近的内置能力或其他已安装 skill 兜底。
- 任何会新增、修改、删除、审阅或验证 agent 行为规则的任务，必须触发 `rule-fix`；范围包括 `rules/*.md`、个人维护 skill 中会约束 agent 行为的规则说明、项目 `AGENTS.md` 和项目 `.codex/rules/`。只读解释或评审只做诊断；只读验证可做隔离验证但不进入写入步骤。

## 中文文字处理 Skill 命中

- 任务涉及写作、修改、润色、审阅或生成中文文字产物时，默认命中 `humanizer-zh`，即使用户没有点名该 skill 或没有明确说“去 AI 味”。
- 中文文字产物包括但不限于：Markdown 文档、README、API 文档、教程文档、需求说明、代码注释、方法说明、字段说明、常量说明、配置说明、错误信息、日志文案、提交信息、PR 描述、Issue 描述、发布说明、UI 文案、产品描述、功能描述和方案描述。
- 用户要求中文内容“更自然”“更像人写的”“人话一点”“更简洁”“更清楚”“减少 AI 味”“去模板化”“去宣传腔”时，必须命中 `humanizer-zh`。
- 仅因用户使用中文交流，不自动命中 `humanizer-zh`；只有当任务产物本身包含人类会阅读的中文文本时才触发。
- 纯代码实现、纯命令执行、纯数据查询或普通中文问答不默认触发 `humanizer-zh`；如果这些任务同时新增或修改中文注释、文档、错误信息、日志、提交信息或 PR 描述，再触发。

## 测试类 Skill 分流

- 测试类请求先按用户目标分流，而不是按关键词或 skill 名称分流；目标是确认既有功能质量时，先按测试规则确定验证对象和环境，不得把任务改写成测试代码交付。
- TDD、测试覆盖、测试框架或本地自动化相关 skill，只在用户把交付物定义为测试代码/覆盖率/测试工程，或用户已经授权代码修改且测试代码属于该修改范围时使用。
- 如果测试任务已由项目规则、本机私有配置或用户长期约定指向某个环境，skill 只能辅助选择验证入口、证据采集和结果整理，不能替代目标环境验证，也不能触发仓库写入。

## 排查类 Skill 命中

- 用户说排查、调查、debug、定位、根因、报错、异常、堆栈、复现、修复 bug、功能突然不可用或类似表达时，默认命中排查类 skill。
- 用户提供 issue 编号、缺陷链接、明确 bug 或复现说明时，优先命中 bug/issue-tracker 类 skill；如果没有可用 bug skill，再命中通用 investigate/debug 类 skill。
- 用户只发截图、链接、现象或“分析一下”时，先判断是否包含异常、错误、数据不一致、接口问题、业务标识或复现线索；普通内容解读、文案理解或资料说明不自动升级成 bug 排查。
- 命中 bug skill 且用户没有明确要求修复或改代码时，只做分析和证据采集，不要直接修改代码，也不要把分析任务升级成实现任务。
- 命中 bug skill 后，在读取 `SKILL.md` 和本机项目配置前，不得调用用户给出的接口、不得临时改用相似地址、不得只凭截图给建议。需要接口证据时，必须按 bug skill 和项目本机配置选择正确层级。
- 如果 bug skill 与 investigate 同时匹配，bug skill 为主 skill；investigate 只能作为追链方法补充，不能替代 bug skill 的触发、证据字段和输出门槛。
- 命中 bug skill 后，接口调用、字段完整性、证据采集、数据时间线、根因证明、测试地址使用、配置持久化和最终输出门槛均按 `bug/SKILL.md` 执行；本文件只负责触发路由、主 skill 选择和不要降级处理，不重复规定字段示例、接口证据格式或配置沉淀策略。
- 非 bug 的日志、API、数据源、命令或工具查询输出按 `evidence-output-rules.md` 执行；涉及 MCP 选择、调用、兜底或连接来源时再叠加 `mcp-output-rules.md`。若同时命中 bug skill，以 `bug/SKILL.md` 为主，`evidence-output-rules.md` 只补充结果展示和脱敏原则。
- 用户点名具体平台或工具时，优先命中该平台的专用排查 skill 和对应规则；例如 OpenClaw 问题同时遵循 OpenClaw 专用规则。
- 用户要求 review、QA、harness、eval、benchmark 或安全审计时，优先命中对应专用 skill；不要把所有问题都降级成通用排查。
- 用户只说“查日志”“排查日志”或“看日志”但没有说明环境时，若项目规则、本机配置或长期约定已有测试环境日志/数据源，默认按测试环境处理，使用项目配置的数据源 MCP 或测试环境数据源；若没有默认环境证据，先确认环境；不能把“日志”本身当成 Grafana 触发条件。
- 只有用户明确点名 Grafana、灰度、线上、生产/online、dashboard、Explore，或已从项目配置/证据确认目标环境是灰度或线上时，才追加 Grafana skill。
- 用户明确指定环境或数据源后，后续日志、数据库和接口证据必须按该边界执行；若已经走错数据源，立即停止错误查询，说明已切换边界，不得继续沿用旧数据源补证据。
- 用户明确指定数据库、缓存、消息队列、搜索引擎等具体数据源时，优先调用对应 MCP；不能因为“查日志”或“排查接口”自动改走 Grafana、浏览器或本地脚本，除非指定数据源不可用并已说明兜底原因。
- 排查类任务需要多个 skill 时，先确定主 skill；追加证据采集、浏览器、测试、安全或观测类 skill 前，必须独立满足该 skill 自己的触发条件，不能只因主 skill 需要证据就越级触发。若用户已明确环境或证据边界，辅助 skill 不能改变该边界。
- 当用户纠正业务术语、指出注释或字段名不对时，必须连同相关判断链一起核对：相邻分支、辅助方法、局部变量、注释和最终结论一起统一术语，不要只改单点。

## 多 Agent 使用边界

- Skill 流程只能服务用户目标，不能把“测试、检查、排查、修复、规则纠偏”升级成无关的 spec、plan、报告或留档交付；是否写文档以项目治理规则和用户明确要求为准。
- 第三方 skill 的流程不能覆盖个人沟通规则；skill 要求写 spec、review 文件或等待确认时，不算真实阻塞，必须先在聊天里用中文给出可审核内容并继续推进非阻塞动作。
- Superpowers 相关 skill 命中时，以 Superpowers 的执行与复核方法为准，包括 `subagent-driven-development`、`dispatching-parallel-agents`、`executing-plans` 及其引用的 reviewer/implementer 流程；但不得绕过安全、隐私、项目治理和用户明确范围，也不得强制生成用户没有要求且对当前目标无用的中间文档。
- 非 Superpowers 场景下，才使用本地保守边界：启用多 agent 前先拆分关键路径和可并行支线；当前下一步所依赖的阻塞任务由主线程自己做，只有边界清晰、可独立推进的支线任务才委派。
- 非 Superpowers 信息排查类委派必须给出具体问题、搜索范围、期望输出和不需要触碰的范围；多个 agent 之间的问题不能重复。
- 非 Superpowers 代码修改类委派必须指定文件或模块所有权，确保写入范围互不重叠，并提醒对方不要回滚他人改动、要兼容并行修改。
- 非 Superpowers 委派后主线程继续推进非重叠工作；只有下一步确实依赖子 agent 结果时才等待。子 agent 完成后，主线程负责快速审查、整合、验证和关闭不再需要的 agent。
- 只有独立上下文、独立会话、独立线程或隔离 worktree 的机制，才能称为真实多 agent 或独立审核；同一对话里的角色扮演只能称为结构化自审，不能包装成上下文隔离的多 agent 协作。
- 多 agent 排查的最终结论必须区分事实证据、推断和未验证假设，并说明每个结论来自哪个 agent、工具、日志、代码或数据源。

## 浏览器自动化会话边界

- 使用带后台 daemon 或 session 的浏览器自动化工具时，优先使用当前任务或当前工具的独立 session，避免多个 CLI 共用默认会话。
- 常规收尾只关闭当前 session，或依赖工具的空闲超时机制；只有用户明确要求清理全部浏览器自动化会话，或已确认没有其他活跃 CLI 依赖时，才使用 `close --all` 之类全局关闭命令。
- 排查发热、高负载或残留进程时，先列出 session、daemon 和子进程关系，再按目标 session 或孤儿 PID 清理；不要关闭普通浏览器或仍活跃的其他自动化 session。

## 推荐与筛选

- 推荐工具、插件、skill、库或框架时，先按资料调研规则收集来源、采用度、维护状态、功能匹配和风险证据，再给推荐层级。
- 没有证据充分的主推荐时，优先说明可用内置能力、已有本地工具或可自建规则/skill，不要默认安装弱证据候选。

## 修改边界

- 不要通过修改官方、第三方或包管理器安装的 skill/plugin 文件来沉淀个人行为偏好；这类文件会被更新覆盖。
- 个人行为约束只能写入 `rules/*.md`、项目 `AGENTS.md`、项目 `.codex/rules/` 或明确属于个人维护的自定义 skill。
- 个人自建或明确个人维护的 skill，唯一来源放在 `ai-agent/skills/<skill-name>/`；各工具的生效目录只保留单个软链接或配置引用。
- 修改 skill/plugin 文件只允许用于用户明确要求维护该自定义 skill/plugin，或临时排查本机问题；不能把它当作全局规则入口。
- 安装、同步或搬迁 skill 前，先检查目标工具实际会扫描的所有 skill 根目录；这些目录来自工具配置、本机 profile 或运行时检测。同名 skill 在同一工具的扫描范围内只能保留一份，避免同一 skill 被重复加载。
- 维护自定义 skill 时，先合并重复、失效或互相打架的规则，再保留最小可执行集合；不要把同一约束散写在多个位置。
- 已有 skill 中的项目级特例不能作为继续添加一次性业务名、字段名、接口名或页面名的先例；新增或修改 skill 规则前同样要做去场景化检查。
- 新增或修改会约束 agent 行为的 skill 规则前，必须先按 `rule-fix` 展示拟改文件、修改理由和不改文件；写入后必须完成隔离验证。
