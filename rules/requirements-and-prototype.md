# Agent 需求与原型规则

- 用户提供原型、截图、网页、文档或链接时，先基于实际输入整理事实，再结合代码或实现判断。
- 如果链接可能需要登录或权限，先按外部链接预检规则判断是否有可复用登录态。
- 原型、代码和用户口头说明冲突时，文档中分开写“原型期望”“代码现状”“待确认项”。
- 按原型、设计稿或 PRD 设计和审阅接口时，必须逐项核对页面可见字段、筛选控件、搜索提示、分页排序、状态、统计值、按钮动作、空态和异常态与接口入参/出参的映射；无法映射时写明代码现状、待确认或 TODO，不得静默省略。
- 复核接口字段时，入参按筛选、搜索、分页、排序、权限上下文和状态切换核对；出参按列表项、元信息、统计、操作可用态、预览资源和提示文案核对。
- 需求文档要能脱离聊天记录独立理解，保留入口、角色、前置条件、交互步骤、状态变化、边界条件和验收口径。
- 用户明确要求落档、生成或更新需求文档，或使用需求采集类工具时，交付物写入当前项目的需求文档库；仅要求分析或说明时，先在聊天中给出可审核内容。
- 写入前先检查 `.docs/` 中是否已有同模块或同义文档；能合并就更新原文档，确实没有合适归属时再新建模块文档，并同步更新模块总览或索引文档。
- 即使 `.docs/` 被 `.gitignore` 忽略，也不能因此跳过写入；最终答复要明确文档已落盘，以及是否被 git 忽略导致未进入版本库。
- 内部链接、账号、token、cookie、测试地址、私有截图、浏览器脚本、登录状态和平台细节遵循安全、隐私与同步边界规则。
