# Agent 资料调研与方案检索规则

## 触发范围

- 用户要求找资料、找方案、主流方案、推荐、选型、同类对比、竞品分析、技术调研、市场调研、资料综述，或询问“现在主流怎么做”“有哪些成熟方案”“哪个更适合”时，必须按本规则执行。
- 用户提出方案类问题时，不要先凭经验直接给完整方案再补查资料；必须先收集足够客观证据，至少完成本规则要求的来源分层后再给结论。证据不足时，只能给临时判断、待验证假设或下一步调研路径。
- 输出前先判断任务类型：事实查询、轻量回答、方案/推荐/选型、高风险决策。事实查询和单链接阅读按用户范围处理；方案/推荐/选型默认按平衡模式处理；只有用户明确要求“只要一句话”“只基于指定来源”“不要调研”，且问题不涉及高风险决策时，才进入轻量模式。
- 普通事实查询、单个链接阅读、明确指定来源的查询、用户要求快速回答的轻量问题，不强制执行完整多源调研；但如果信息可能近期变化，仍要按可用工具核验。轻量问题仅指不要求推荐、选型、方案判断、主流判断、优劣对比或决策建议的事实性问题。
- 用户明确要求“简单说”“不用查太多”“只看官方文档”时，以用户范围为准，但结论中要说明来源范围被收窄。同一请求同时包含这类收窄表达和“方案、推荐、选型、主流、对比、最佳实践、哪个更适合”等决策表达时，仍按方案/推荐/选型任务处理；收窄只影响输出篇幅和来源范围，不取消最低证据要求。
- 高风险主题不得因用户要求“简单”“快速”而跳过多源核验；可以压缩结论，但必须保留关键证据、风险和适用边界。

## 多源检索要求

- 不得只凭单次网页搜索、单篇文章、单个平台结果或模型记忆给出“主流方案”“推荐方案”“最佳实践”结论。
- 平衡模式下，涉及外部推荐、主流判断、用户可能投入明显时间或金钱、生产或技术选型、工具/skill/plugin 推荐、技术栈或工作流变更时，至少覆盖 3 类证据和 3 个相互独立的有效来源；高影响决策、付费工具、生产架构、技术栈迁移、安全/合规/金融/医疗等高风险主题至少覆盖 5 类证据和 5 个相互独立的有效来源。仅覆盖来源类别但缺少足够独立来源时，不得给强结论。
- 低风险内部判断可以基于本地项目上下文、一手/官方来源和明确的不确定性收敛，不强制扩展成完整外部调研；但不能使用“主流”“最佳”“成熟首选”等强判断。
- 用户强制要求极简输出且仍包含推荐、选型或方案判断时，至少核验 2 类证据，并明确说明“这是快速结论，证据范围有限”。
- 同一域名、同一厂商材料、转载内容、同一作者矩阵、搜索结果摘要、聚合页和同源引用链，不重复计为多个独立来源。疑似软文、返佣导购、广告内容或厂商投稿只能作为弱证据，并在结论中降权说明。
- 证据类别优先从以下来源选择：
  - 官方文档、官方博客、标准规范或项目主页。
  - 采用度数据，例如 GitHub stars、forks、release 活跃度、包管理器下载量、安装量、生态榜单、同类排名。
  - 代码与开发者生态，例如 GitHub 仓库、issue、PR、release notes、示例项目、awesome list。
  - 社区讨论，例如 Reddit、V2EX、HN、Stack Overflow、微博、公众号、小红书、Twitter/X、论坛评论。
  - 近期文章、评测、迁移案例、事故复盘、性能或成本对比。
  - 本地项目上下文，例如现有代码、依赖、团队约束、部署环境和历史决策。
- 遇到站点无法访问、工具失败、需要登录、反爬、付费墙、读取失败、结果为空或渠道失效时，不得直接用剩余少量结果下结论；必须先尝试换源补证。尝试 2-3 种替代路径后仍不足时，停止扩展检索，降级结论并说明缺口。
- 换源补证至少包括两类动作：更换检索关键词或语言、更换搜索/读取工具、更换来源类型、查找官方替代页、查找 GitHub/包管理器/社区/新闻/存档等公开证据。
- 如果某类关键证据仍不可用、需要登录、渠道失效或检索不到，要说明已尝试的替代渠道和仍缺失的证据；不能把未查到的来源伪装成已验证，也不能只写“未查到”就完成调研。

## 工具优先级

- 已安装 `agent-reach` 时，资料调研优先把它作为多渠道入口，而不是只用通用搜索。根据任务风险先确定证据类别数量，达到所需证据后停止扩展；不要为凑渠道继续搜索。
- 根据任务组合使用 Exa、网页阅读、GitHub、RSS、公众号、V2EX、微博、视频字幕、社媒或其他可用渠道。某个渠道不可用时，优先用同类公开渠道替代，而不是直接放弃该证据层。
- 技术选型、主流方案、推荐方案类结论必须至少核验一个官方或一手来源，包括官方文档、项目主页、标准规范、官方博客或官方仓库；无法访问时必须说明原因，并用可信的一手替代来源补充。
- 技术选型默认必须同时检查公开代码仓库或发布记录，以及包管理器或生态采用度；至少记录 stars、release 活跃度、最近维护时间、下载量或同类排名中的关键数据，并标注查询日期或“截至本次查询”。没有 GitHub 或包管理器数据时，改用官网更新记录、应用商店、社区活跃度、客户案例等替代证据。不能只引用博客观点。
- 主流方案或生产选型至少补充社区反馈、真实案例、迁移经验、事故复盘、性能/成本评测或企业采用证据中的 2 类；高风险主题至少补充 3 类。没有这类证据时，只能给候选方案，不能下“主流最佳方案”结论。
- 判断“主流”不得只依赖搜索结果页、博客合集或单一内容平台；同一搜索引擎、同一媒体站、同一社区平台下的多篇内容不能算作多类证据。
- 中文语境、国内产品、消费口碑、国内工具推荐或平台内容，除非用户明确限定来源，否则必须优先覆盖中文来源，不得只看英文结果或只用英文资料下推荐结论。
- 中文产品、消费口碑和国内工具推荐至少覆盖 3 个互不隶属的中文来源，其中至少 2 个必须是用户口碑渠道。官方文档、厂商公告、媒体评测和导购榜单不计入口碑渠道。
- 用户口碑渠道包括但不限于知乎、V2EX、小红书、微博、B站评论或字幕、豆瓣、少数派评论、什么值得买、电商评价、应用商店评价、黑猫投诉、论坛帖子、公众号评论或转载讨论。
- 中文产品和国内工具检索时，能确定的中文名称、常见别名和竞品中文名都要纳入检索；风险关键词至少覆盖 2 类，例如“差评、投诉、退款、售后、隐私、停服、翻车、避坑、收费、故障”。
- 如果公众号、社媒、论坛或评论区不可访问，必须补查至少 2 个其他中文口碑渠道；仍不足时，不得给出强推荐，只能说明证据不足并列候选。
- 英文技术主题默认补充英文官方与 GitHub 来源。
- 如果用户点名特定平台或 skill，先按 `skill-rules.md` 加载对应 skill，再用本规则约束证据数量、来源分层和输出。
- 不为追求数量盲目安装新 skill、插件或 MCP。现有工具能覆盖证据需求时，优先使用现有能力；推荐新增工具前必须按本规则收集采用度、维护状态、来源可信度、功能匹配和风险证据，并按 `skill-rules.md` 检查触发和修改边界。证据不足时不得建议安装新 skill，只能列为候选或建议自建规则。

## 分析与输出

- 先区分事实、推断和建议。事实要能追溯到来源；推断要说明依据；建议要说明适用条件和不适用条件。
- 推荐类结论必须分为“主推荐”“候选”“不推荐/暂不推荐”或等价层级。主推荐必须有采用度或明确约束下的适配证据。
- 对比多个方案时，至少覆盖：使用场景、采用度、维护状态、集成成本、主要风险、适合/不适合的边界。
- 如果证据不足以选出主推荐，或达不到当前任务的最低独立来源数量，应明确说“没有证据充分的主推荐”，再给可继续验证的候选或本地可行方案，并说明已尝试的来源类型、失败或缺失的关键来源、继续验证的优先渠道。
- 当官方来源、采用度数据、社区/案例证据任一关键层缺失时，禁止使用“最佳”“业界主流”“成熟首选”等强判断，只能表述为“当前证据下较可行的候选”。
- 消费口碑和国内工具推荐必须单列“负面反馈/主要争议/不适合人群”；没有查到负面反馈时，要说明已检索的负面渠道和关键词。
- 回答中要简要说明检索覆盖了哪些来源类型；不需要贴长日志或原始搜索结果。关键链接、数据和日期要保留。
- 方案、推荐和选型回答要先交代证据边界，再给建议；如果有内容来自模型推断、本地经验或类比判断，必须明确标为推断，不能伪装成调研事实。
- 不输出大段未经筛选的资料堆砌；只保留能改变判断的证据和与用户决策有关的差异。

## 时效与复核

- 涉及最新版本、价格、政策、API、库生态、维护状态、流行度、公司/产品状态、社区评价时，必须联网或调用可用工具核验当前信息。
- 同一结论如果多个来源冲突，优先采用官方和一手数据；社区评价作为风险和体验证据，不替代官方事实。
- 引用下载量、安装量、star、发布日期等会变化的数据时，要说明查询日期或使用“截至本次查询”这类表述。
