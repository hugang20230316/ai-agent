# Agent 测试与验证规则

- 用户目标是确认既有对象是否满足预期但未说明环境时，先确认使用本地、测试环境还是其他环境；如果项目规则、本机私有配置或用户长期约定已说明默认环境，则直接按该环境执行，不再追问，也不能改成本地验证。
- 测试类请求默认表示对既有功能做环境或行为验证，不表示修改仓库、补测试代码或创建测试基础设施；只有用户明确把交付物定义为测试代码、覆盖率或测试工程时，才进入测试代码编写路径。
- 执行测试任务前先做动作分流：先判断用户要的是运行中系统的行为证据、离线检查证据，还是测试代码交付；不能用一种证据替代另一种目标，也不能因为某种验证更容易执行就改写用户目标。
- 测试规则只限定动作顺序，不给任务贴固定标签：先用无副作用手段取得可观察证据，再根据证据说明是否需要修复；没有修复授权前，写入只允许落在测试产物或临时产物，不改被测对象。
- 测试中发现疑似问题时，先区分已经复现的失败、尚未验证的推断和预期内行为；只有已经复现失败并且用户授权继续修复时，才进入代码修改。
- 用户明确要求测试环境、线上或发布后验证时，验证对象必须是对应环境；本地构建、本地单测和静态检查只能作为补充，不能替代环境验证。
- 测试环境接口验证必须使用项目配置、项目规则或对应 skill 指定的层级；不要改用相似环境地址、相邻层级或前端地址补证据，除非用户明确要求验证那些层。
- 项目测试环境的具体地址、账号、token、默认写入授权和代理绕过方式属于项目或本机私有配置；全局测试规则只描述如何使用这些配置，不能写入具体项目地址或环境细节。
- 如果项目本机私有配置已明确记录测试环境允许写入、修改、删除和回滚验证，则该配置视为用户对该项目测试环境的持续授权；按配置直接执行，不要为普通测试数据写入、删除或回滚反复询问。
- 环境验证要区分只读发布确认、接口存在确认、功能行为验证和写入型端到端验证；不能把 Swagger 可访问、接口存在或本地构建成功当成功能修复已在目标环境生效。
- 只读查询、静态检查和无副作用验证可以直接执行；写入、删除、迁移和真实数据修改必须先确认。
- 用户明确下达“发布”“同步”“部署”等执行指令，且操作范围来自当前仓库已配置的发布 skill、脚本或默认范围时，该指令本身就是发布授权；发布流程内必要的读取本机配置、读取凭据、创建 tag、等待流水线、同步默认应用和只读复查不再二次确认。
- 发布前必须先检查本次提交或待发布 tag 的变更文件、提交信息和可识别组件，判断实际影响范围；默认发布范围只能作为兜底，不能替代变更影响判断。
- 若变更影响的组件与默认发布应用不一致，先按仓库配置映射到对应应用发布；映射不清或可能漏发时，先向用户说明变更范围和建议发布范围，再继续执行。只有目标范围超出配置、需要新增或修改凭据、涉及真实业务数据或破坏性动作，或指令存在互斥解释时，才等待用户确认。
- 发布、同步或部署只有在目标范围超出当前配置、需要新增或修改凭据、需要写入真实业务数据、包含破坏性动作，或用户指令存在互斥解释时，才停下确认最小必要问题。
- 如果平台沙箱要求对已授权发布流程的固定入口做命令审批，只把它当作工具运行权限处理；不要把审批措辞包装成“是否发布”的业务确认，也不要为同一发布流程重复申请等价权限。
- 真实数据写入测试前，先准备可执行回滚方案；执行前展示影响范围和回滚方式。
- bug 修复验证必须说明是否覆盖历史存量数据；没有迁移脚本、数据修复脚本或查询兜底时，默认只能证明后续流程修复，不能说历史脏数据已经自动恢复。
- 测试结论必须说明验证命令、验证对象、实际结果和未覆盖风险。
- 涉及跨服务链路、消息队列或异步处理时，验证结论必须串联入口接口、实际生效配置，以及消息队列、数据库、日志中的至少两个后端证据源；无法覆盖时说明缺口。
- 异步任务类问题按阶段拆分验证：任务创建、消息被消费、回调到达、状态落库、查询接口返回；不能把任一阶段成功当作全链路成功。
- 构建或测试失败时，先区分环境问题、依赖问题、权限问题和代码问题，不把它们混成一个结论。
- 同一任务内已确认某验证命令因缺依赖、缺权限或外部条件不可解而失败，并已改用替代验证后，记录该失败命令、原因和替代验证；不要再次运行同一失败命令，除非环境已明确改变。
- 本地服务验证时，先确认服务实际监听地址和端口，再发请求；不要凭历史端口试探。
- 区分接口服务地址和可视化预览地址：POST 接口不能直接当作 Web preview 交付；用户需要查看完整接口返回时，提供原始响应文件或只读静态预览页，并先验证预览页及其依赖的 JSON/资源都返回 200。
- 给用户用于查看测试结果的本地服务或预览服务时，不要在用户查看前擅自停掉；如需释放端口，先等待用户确认或明确说明服务已停止。
- 测试产物默认写入系统临时目录；日志、JSON 响应、HTML 预览、截图和缓存不要放进项目仓库。若确需短暂放入仓库目录，必须在最终答复前清理并确认 `git status --short` 不包含这些产物。
- 遇到 `sudo` 或其他交互式命令时，先用非交互方式判断能否直接执行；若必须输入密码或进入 TTY 交互，立刻把它标记为用户阻塞或改走无交互方案，不要在宣称“继续处理”的同时结束本轮。
- 测试过程中产生的日志、截图、缓存、临时文件和平台差异命令遵循安全、隐私与同步边界规则。
