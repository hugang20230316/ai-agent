# 代码合规审查

这是代码改动后的合规门禁，只审最终 diff 是否符合当前仓库规则、公共编码规则和验证规则。它不是需求验收、spec review 或功能验收。

本 skill 不定义新的编码标准，不复制具体红线。具体标准只从当前项目规则、`coding-rules.md`、`testing-rules.md` 和 `skill-rules.md` 读取。

## 必读规则

- 先读最近的项目 `AGENTS.md` 或等价项目规则入口。
- 审查 diff 前必须读取公共沟通、编码、测试和 skill 规则。
- 改动涉及凭据、外部调用、上传、部署、公司项目边界或本机私有配置时，再读取安全与隐私规则。

## 审查目标

- 每次执行前先确定目标仓库和审查对象，不得把当前 agent 所在的个人规则仓库、skill 源仓库、工具仓库或上层 home 目录当作默认审查目标。
- 用户明确给出仓库、路径、分支、commit、GitHub/GitLab 提交链接、PR 或 MR 时，以该对象为准。
- 主动触发但未指定目标时，先选择用户最近明确讨论、正在实现或上下文核心需求对应的业务/公司代码仓库，并审它的未提交改动；其次才考虑当前工作目录所在业务/公司 Git 仓库。找不到业务/公司仓库或候选不唯一时，停止并让用户指定目标。
- 个人规则、skill、工具或上层 home 仓库不能作为主动触发的兜底目标；只有用户明确要求审查这些仓库，或本轮原始任务就是维护这些仓库且没有未完成的业务/公司代码任务时，才可选择。
- 自动触发时，目标是本轮代码改动所在仓库；跨仓改动必须逐仓审查，不得把多个仓库的 diff 混成一次结论。
- 用户明确要求只审不改，或目标是历史 commit、PR/MR、远端提交链接时，默认只读审查。只有用户要求把问题修到当前工作树，才进入修复步骤。

## 触发边界

- 本轮创建、修改、生成、修复、重构或格式化了代码时，最终答复、提交或创建 PR 前必须执行。
- 用户要求对当前改动、未提交代码、指定 commit、GitHub/GitLab 提交、PR/MR 做 `review-coding`、审查、审核、修复或按规则修复时必须执行。
- 需求或 spec 是否实现属于另一类 review；本 skill 只用需求上下文确认授权范围，避免误撤用户想要的行为。
- 纯只读解释或代码浏览不自动触发，除非用户要求按规则审核。

## 提交记录审核

- 审核指定 commit 或提交链接时，先定位对应仓库，再用该提交自身 diff 确认它引入的修改；merge commit 默认按第一父提交比较，除非用户指定父提交。
- 不能只按历史提交快照下结论。必须再查看当前最新代码中同一批受影响文件：比较该提交到当前 `HEAD` 的变化，并检查这些文件是否还有工作区未提交改动。
- 结论必须区分：问题由所选提交引入、问题在最新代码仍存在、问题已被后续提交改变或移除、最新工作区改动影响判断。

## 审查修复循环

重复执行完整循环，直到 diff 达标或出现真实阻塞：

1. **审查**
   - 查看 `git status --short`、`git diff --stat` 和每个代码改动文件的最终 diff。
   - 先确认本轮授权写入范围，区分当前任务改动和已有用户改动。
   - 按编码规则检查无关格式化、顺手重构、新抽象、新依赖、成员顺序漂移、命名漂移、越权契约改动和过度封装；先区分当前目标必需的同一判断链同步修改和顺手扩大范围。
   - 按编码规则检查本轮新增或改写的局部变量；只被下一次调用、下一行条件或单个表达式使用，且仅承接成员访问、空白修剪、默认值、布尔判定、类型转换或参数转发的变量，必须内联。只有能证明它避免重复求值、副作用或显著表达业务分支时才保留，并在审查结论说明证据。
   - 审查本轮 diff 时，必须列出新增或改写的关键分支条件及其注释状态；缺少就近业务注释时，review-coding 不通过。
   - 按编码规则对本轮新增、改写或同一判断链受影响的人类可读文本做 diff 级审查；范围包括代码注释、字段/常量/配置项说明、DTO/接口说明、`Display`/Swagger/OpenAPI/校验提示/错误信息。
   - 对本轮触碰的字段、DTO、接口和对外文本，先用字段名、实体属性、映射、同类 DTO、枚举和已有文档追溯项目主术语；用户口头称呼、页面临时文案或一次性需求词只能作为输入线索，不能覆盖仓库已有基础语义。
   - 注释和对外文本只判三类问题：缺少必要业务含义或判断原因、无必要新增且只复述条件或方法链、用使用场景或同义词漂移替代基础语义。
   - 审查结论必须逐项写明局部变量、helper、人类可读文本、字段/DTO/契约术语的检查结果；本轮未涉及的项也要标明未涉及，不得用“已检查”概括。
   - 对本轮新增、抽取或签名变化的私有/helper/辅助方法，必须逐个列出方法名、调用点数、参数数、方法体类型、保留证据和保留/内联结论；漏列新增 helper，或只写“已检查”，视为审查失败。方法体类型按文本拼接、单行调用、默认值兜底、参数搬运、稳定计算规则、复杂业务步骤等口径判断。
   - 单行调用、简单表达式、取默认值、取尺寸/长度/数量、空值兜底、类型转换、元组搬运或纯参数转发的 helper，即使多处调用也默认不保留，除非能证明方法名表达稳定业务概念且方法体承载非显然规则。
   - 审查 helper 时必须统计方法体有效代码行数和调用点；有效代码 1 到 5 行且调用点不超过 5 个的私有/helper/辅助方法，review-coding 不通过并要求内联。
   - 检查 helper 的方法注释：空 `param`、空 `returns`、只复述方法名或只翻译代码的注释，不能作为保留 helper 的证据；若 helper 本身不该存在，优先内联而不是补注释。
   - 对本轮新增的私有类型或语言等价结构，确认不是只服务 helper 的承载壳。
   - 结构合规和构建、测试、运行结果分开判断；功能验证不能覆盖结构审查失败。

2. **修复**
   - 阻塞问题在授权范围内可修复时，直接修。
   - 优先内联无价值局部变量和 helper、移除 helper-only 私有类型、压缩参数搬运式封装、撤回顺手重构；简单变量或 helper 不要用补注释、改名或再加一层包装来补偿。只读审查或未获修复授权时，只列为结构合规问题并说明应内联，不直接改工作树。
   - 注释或对外文本问题可修复时，优先恢复项目已有基础术语；缺注释按当前文件既有注释密度和标点风格补短注释，无必要新增注释直接删除。注释只写业务含义、判断原因或边界，不用注释复述代码。
   - 保留用户已有改动。必要修复超出授权时，停止并说明具体阻塞。
   - 对本轮合规修复运行最小相关验证，证明触碰行为没有被破坏。

3. **再审查**
   - 每次修复后重新读取最终 diff，不依赖上一轮记忆。
   - 仍有可修复阻塞问题时，继续下一轮修复。

## 收口条件

- 只有代码合规审查通过，且本 skill 造成的修复已有对应最小验证，才能收口。
- 只有缺授权、环境不可用、凭据/外部审批缺失，或当前范围内不可修复时，才算阻塞。
- 仍有可修复合规问题时，不得发送完成类最终答复。
- 任何含本轮代码改动的最终答复，正文开头必须先给出 review-coding 证据块，缺这块视为门禁未执行、未收口，禁止发送。证据块固定一行四项，格式如下：

```
review-coding: 第N轮 | 改动文件: a.cs,b.cs | git diff --check: 通过 | 无关改动: 无
```

- 证据块以"本轮是否改过代码"这一客观事实触发，不以改动大小或主观必要性触发；改动行数再少也必须带。四项必须据实填写：`第N轮`为审查修复循环次数，`改动文件`为本轮实际改动文件名，`git diff --check`填实际运行结果，`无关改动`填"无"或具体已撤回项。无法运行 `git diff --check` 时如实写明原因，不得留空或编造"通过"。

## 最终汇报

最终汇报保持简短，只写证据：

- 审查/修复循环次数；
- 本轮改动文件；
- 已修复或仍阻塞的结构合规问题；
- 局部变量、helper、人类可读文本、字段/DTO/契约术语和无关改动检查结果；
- 合规修复的验证命令和结果；
- 剩余风险。
