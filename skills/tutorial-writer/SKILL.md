# Tutorial Writer — 技术教程写作专家

## 核心目标

教程的目标不是把步骤写完整，而是让读者真正理解、少走弯路、能迁移到相邻场景。

不要奖励时间切片、安装流水账、机械验证，也不要把教程写成资料综述、审稿表或评分打卡。读者只关心：为什么学、卡在哪里、这东西怎么工作、能不能用到自己的问题上。

## 最小流程

### 0. 写前校准

写教程前先确认读者画像和目标深度。至少判断：

- 读者水平：初学者、资深开发但不熟该技术、熟悉该技术想深入。
- 目标场景：入门理解、项目落地、生产运维、面试准备、深度原理。
- 技术上下文：语言栈、部署环境、是否涉及集群、配置、性能、安全或排障。

如果用户已经给出这些信息，直接记录假设，不要重复询问。如果缺失且会影响教程取舍，先问一个最小必要问题。

不要把主题必备知识点外包给用户判断。资料和主题性质已经表明属于核心的内容，作者必须主动覆盖；没讲就是遗漏，不能解释为“用户没有要求”。只有可选深入点才列出来问用户是否需要，例如源码细节、特定语言 SDK、特定部署平台、专项性能压测或非常细的实现原理。

当用户只给出读者画像或目标场景时，作者应自行用资料覆盖地图补齐必讲项；只向用户确认可选深入方向。

### 1. 确认必要依据

写作或审核前先确认必要依据。资料采集只记录事实、来源和边界，不分析、不提炼核心，不替代教学判断。

按任务轻重决定采集规模：单段解释、补一个例子、普通 review，只确认关键事实来源；入库、发布、深度教程、高风险主题，再整理完整资料证据包。

完整证据包按以下类别整理：

- **官方事实**：官方文档、规范、源码、release notes、API 行为、版本差异、限制。
- **主流解释**：优秀教程、课程、书籍章节、公开讲义、权威文章如何解释这个主题。
- **常见误区**：社区讨论、issue、问答、踩坑文章里反复出现的问题。
- **争议和边界**：来源冲突、版本差异、适用条件、不适用场景。
- **主流同类**：工具、框架、平台、产品或方案类教程要检查是否存在主流同类；有则记录同类对象、采用度证据、官方定位、典型场景、已知不足和差异来源。
- **知识覆盖地图**：列出主流资料反复覆盖的主题，并标注必讲、建议讲、可选、暂不讲。资料采集员只列地图和依据，不做教学取舍。

正文不要展示资料全貌，只吸收对读者有用的结论。资料清单可以放在依据、附录或审查记录里。

如果技术点可能变化，必须查最新资料。资料不足时标注边界，禁止用“最佳实践”“主流做法”“高频面试题”包装未核验内容。

### 2. 核心教学判断

基于资料先回答下面 6 个问题。重要教程再从事实准确性、教学有效性和结构节奏三个视角复核。

1. 这个主题真正解决什么问题。
2. 读者最容易误解的 1-3 点。
3. 必须讲透的机制是什么。
4. 哪些内容应该删除或放附录。
5. 最小有效示例是什么。
6. 学完后读者应该能迁移到哪类问题。

如果主题涉及工具、框架、平台、产品或方案，并且存在主流同类，还要回答：它和主流同类的差异、擅长、不足分别是什么，证据来自哪里。

如果主题是开发者技术教程，还要基于资料覆盖地图判断哪些模块属于必讲。常见维度包括核心模型、关键机制、配置与参数、生产部署或集群、高可用与故障处理、监控排障、安全边界、同类对比、面试或能力点。不是每篇都要全写，但被资料和主题性质共同证明为核心的模块不能缺失。

正式写作或入库审核前，至少要能回答这 6 个问题。

### 3. 写作或改写

正文围绕核心教学判断展开。操作步骤只能服务理解，不能撑起主体。

优先写：

- 问题背景和学习动机。
- 核心概念和概念关系。
- 机制、边界和误用风险。
- 真实例子、反例、常见误区。
- 能迁移到相邻场景的判断方法。

压缩或移走：

- 安装、部署、点击、等待输出、确认成功。
- 固定时间切片，例如“0-5 分钟”“5-10 分钟”。
- 只证明“我做过”的逐项验证。
- 与核心理解无关的资料堆砌。

### 4. 合并审核

默认合并审核事实、教学、结构、低价值内容和读者体验。只有入库、发布、重要教程，才拆成多个独立视角审核。

审核不能只给分。每个发现必须指出具体段落、具体问题和具体改法。

## 硬门禁

触发任一项即不达标：

- 关键事实没有可追溯依据。
- 核心教学判断不是基于资料。
- 关键技术事实无来源、过时，或与官方/一手资料冲突。
- 未确认读者画像和目标深度，导致讲解层级明显错位。
- 资料覆盖地图显示的必备知识点缺失，却没有明确说明取舍原因。
- 存在主流同类产品或方案，却没有基于证据说明差异、擅长和不足。
- 机械步骤、时间切片、安装部署、逐项打卡验证占主体。
- 核心概念讲不清，读者只能照抄不能理解。
- 高风险或易误解内容没有边界、反例或误用提醒。
- 审核只给分，不指出具体段落、具体问题和具体改法。

## 读者效果门禁

不要要求固定出现“原理”“概念关系”“反例”“举一反三”这些标题。形式自由，但成稿必须让读者做到：

- 能说清为什么这样设计。
- 能区分相似概念。
- 能识别错误用法。
- 能迁移到一个相邻场景。

## 删减门禁

发布或入库前，简单说明取舍：

- 删除了哪些机械步骤。
- 哪些资料、命令或扩展内容移到附录。
- 保留了哪些核心解释。
- 为什么这些保留内容对读者有用。

每轮返工只处理最影响阅读的前三个问题。不要为了把分数从 88 刷到 92 无限打磨。

## 教程类型强度

| 教程类型 | 重点 | 机械步骤上限 |
| --- | --- | ---: |
| 入门概念 | 概念模型、类比、误区 | 15% |
| 实战项目 | 设计取舍、关键路径、验证、排错 | 35% |
| 工具上手 | 最短可用路径、常见坑、适用边界 | 50% |
| 开发者技术 | 读者层级、核心模型、配置、生产边界、排障、面试能力点 | 25% |
| 面试题解析 | 题源证据、思考框架、误区、边界、追问 | 接近 0 |
| 深度原理 | 来源、机制、争议、抽象能力 | 按主题判断 |

实战项目可以保留较多操作，但每组步骤必须解释设计意图。工具上手可以保留最短路径，但不能写安装流水账。

开发者技术教程必须根据读者层级调节讲法：初学者先建立直觉和术语；资深开发但不熟该技术时，少讲通用开发常识，多讲该技术独有模型、配置取舍、生产坑和迁移判断；熟悉该技术想深入时，重点放在边界、机制、性能、故障和源码或协议层。

## 面试题解析

面试题解析保持轻量，只确认三件事：

- 不能凭空说“高频”，必须有主流题源或能力点证据。
- 答案必须有官方、规范、源码、权威教材或成熟资料支撑。
- 不能只给背诵答案，必须讲考察点、常见误区和至少一个追问方向。

可选结构，按需保留，不强制套标题：

```markdown
## [问题]

### 考察点
[面试官真正想确认什么能力]

### 简短回答
[适合开口第一分钟的答案]

### 展开回答
[原理、机制、边界、场景]

### 常见误区
[容易答错、答浅、答偏的点]

### 追问
[至少一个常见追问方向]

### 依据
[官方/规范/源码/权威资料/主流题源]
```

## 内容原则

### 先讲为什么，再讲是什么

读者跳过教程的常见原因是：不知道为什么要学。章节开头先建立问题或动机，再展开概念。

```markdown
<!-- 差 -->
## Redis 分布式锁
Redis 分布式锁使用 SET key value NX PX 命令实现...

<!-- 好 -->
## Redis 分布式锁

秒杀活动中，1000 个请求同时扣减库存，数据库事务来不及加锁，
最终库存扣成了负数。单机 synchronized 没用，因为服务部署了 3 个实例。

分布式锁要解决的是：多个进程同时操作同一份共享资源时，如何让关键操作排队。
```

### 架构、系统、工具教程先讲差异

读者最先关心的通常不是目录，也不是文件职责，而是：

1. 这套东西和我平时的做法差在哪。
2. 它到底多了什么约束或能力。
3. 我什么时候该用，什么时候不该用。

禁止一上来就按目录、文件、模块逐个展开。

### 产品、工具、方案要讲同类差异

如果主题有主流同类产品、工具、框架、平台或方案，正文应说明它和同类的差异、擅长、不足和适用边界。

比较必须有证据，不能凭印象写“更轻量”“更强大”“更适合团队”。可用证据包括：

- 官方定位、文档、release notes 或规格说明。
- GitHub stars、forks、release 活跃度、包下载量、应用安装量、生态榜单或同类排名。
- 社区讨论、issue、迁移案例、事故复盘、性能或成本评测。

证据不足时，只能写“当前资料不足以判断”，不能强行下结论。

### 举例必须来自真实痛点

举例的标准是读者看到后能说“我遇到过这个”。

| 差举例 | 好举例 |
| --- | --- |
| 用 A、B、C 代替真实对象 | 用订单、登录、定时任务、库存扣减 |
| 数字随意填写 | 给出贴近场景的规模，如峰值 QPS 5000 |
| 只有成功路径 | 同时展示失败场景和处理方式 |
| Hello World 级别 | 包含并发、失败、边界或维护成本 |

### 复杂概念分层解释

复杂概念先建立直觉，再给准确机制，最后用最小示例落地。

```markdown
**类比**：去超市买 10 件商品，你可以来回跑 10 趟，也可以列好清单一次性取完。

**机制**：Redis Pipeline 将多个命令连续发送，减少网络往返次数。它不是事务，不保证命令整体成功或回滚。

**示例**：只保留说明 Pipeline 与普通命令差异所需的最小代码。
```

### 图表服务理解，不服务装饰

如果关系、流程或结构用文字超过 3 句话仍说不清，优先用图。图表必须承载理解，不要为了“丰富”而强加。

图表应贴近读者真实工作中会看到的东西：

- 数据库设计：ER 图、表结构、查询计划。
- 微服务/分布式：服务拓扑、调用链、部署图。
- 性能优化：优化前后数据对比、瓶颈标注。
- API 设计：请求/响应、状态码和错误处理。

### 代码示例最小可运行

- 只包含说明当前概念必需的代码。
- 关键注释解释“为什么这样写”，不要复述代码做了什么。
- 复杂代码用渐进式演进，但不要为了形式拆版本。
- 需要对比时展示错误写法和正确写法，并说明差异。

## Gate 评分

只有 `/tutorial-writer gate`、入库、发布、终审才必须评分。普通 review 只列问题和改法。

需要评分时按 100 分制：

| 维度 | 权重 | 审查重点 |
| --- | ---: | --- |
| 资料依据与事实准确性 | 25 | 来源是否充分，版本是否清楚，事实是否可靠 |
| 核心提炼质量 | 20 | 是否抓住核心概念、主流关注点、误解、边界和必要的同类差异 |
| 教学有效性 | 25 | 路径是否顺，例子是否真实，读者是否能理解而不是照抄 |
| 实用闭环 | 15 | 是否能迁移到真实场景，关键判断是否可核验 |
| 编辑质量与节奏 | 15 | 结构是否清楚，语言是否自然，是否避免流水账和模板腔 |

达标线：

- 总分 >= 85。
- 资料依据与事实准确性 >= 90。
- 核心提炼质量、教学有效性各 >= 85。
- 任一硬门禁触发则不达标。

需要评分时，只按最影响阅读的前三个问题返工；不要为分数反复打磨。

## 自检清单

完成教程后确认：

- [ ] 关键事实有可追溯依据，重要教程已建立资料证据包。
- [ ] 已确认读者画像、目标深度和必要技术上下文；缺失项已做合理假设或最小询问。
- [ ] 已建立知识覆盖地图，并区分必讲、建议讲、可选和暂不讲。
- [ ] 必讲知识点没有遗漏；可选深入点才询问用户是否需要。
- [ ] 已完成核心教学判断，且判断来自资料。
- [ ] 正文没有被机械步骤、时间切片或安装流水账占满。
- [ ] 核心概念、边界、误区和迁移方向清楚。
- [ ] 存在主流同类时，已基于证据说明差异、擅长、不足和适用边界。
- [ ] 高风险或易误解内容有反例、边界或误用提醒。
- [ ] 关键技术事实有来源，版本和适用范围清楚。
- [ ] 图表、代码、练习和验证都服务理解，不是装饰或打卡。
- [ ] 已做删减门禁检查。
- [ ] 审核指出了具体段落、具体问题和具体改法。

## 适用场景

- `/tutorial-writer` — 从零开始写一篇教程。
- `/tutorial-writer review` — 审查并改进现有教程。
- `/tutorial-writer gate` — 入库或发布前做质量门禁、评分和终审。
- `/tutorial-writer example` — 为某个概念补充更好的举例。
- `/tutorial-writer explain` — 重新解释某个概念。
