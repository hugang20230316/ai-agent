# 场景 11：关联项目必须代码对照

## 历史失败

需求采集时只整理截图或原型事实，没有结合当前项目代码判断已有能力、缺口、冲突和外部边界。后续才发现有些元素已有 DTO、接口或异步回流，有些则完全没有代码锚点。

## 用户输入

请采集这个原型页的详细需求。这个页面和当前仓库有关，要求结合现有代码查漏补缺，说明哪些已有、哪些缺口、哪些需要新增接口或字段。

## 必须覆盖

- 先列出页面可见的关键元素、状态、动作和数据项。
- 做需求-代码对照，至少覆盖入口/动作、展示数据、状态来源、跨服务能力和旧结论复核。
- 代码锚点不能只找同名 Controller；需要检查 DTO、模型、枚举、Service、消息、消费者、外部适配或旧文档。
- 找不到代码锚点时，写 `[待确认][无代码锚点]`、`[推断][新需求]` 或 `[推断][外部边界]`，不能删除原型需求。
- 原型期望、代码现状、缺口和本轮判断要分开写。

## 失败判定

- 只输出原型需求，没有代码对照。
- 只写“未发现接口”，没有继续检查 DTO、模型、枚举、异步消费者或外部适配。
- 把代码没有实现的原型元素静默删除。
- 把外部系统或包装层误写成本仓库已完整实现。
