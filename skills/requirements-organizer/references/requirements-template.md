# 需求输出模板

按下面模板整理结果。可以根据模块实际情况删减无关小节，但不要省略“需求范围”“详细需求”“验收标准”“待确认问题”。
如果材料明显不足，再补一个“信息缺口”小节。

## 标注规则

- 用 `[明确]` 标识原型、标注、文案里已经明确给出的结论
- 用 `[推断]` 标识基于页面结构、交互路径或常见产品模式推导出的结论
- 用 `[待确认]` 标识证据不足、需要补充确认的内容

## 推荐模板

### 小需求开发版

适用于一个小功能点，优先给开发直接看。

```md
# <功能名>

## 1. 功能概述
- 做什么：
- 入口：

## 2. 交互与表现
- [明确] ...
- [明确] ...

## 3. 规则
- [明确] ...
- [推断] ...

## 4. 异常/边界
- [待确认] ...

## 5. 待确认问题
- ...
```

小需求默认优先使用这个模板，不强制展开完整 PRD 结构。
额外限制：

- “待确认问题”默认最多 3 条
- 只能写阻塞开发实现、联调或验收的问题
- 没有证据的按钮、字段、状态，不要写进正文，也不要写进待确认

## 补充要求

- “详细需求”要写成可以交付设计、研发、测试继续使用的表达，不要只写“这里有一个按钮”
- “验收标准”要尽量可测试、可判定，避免空泛描述
- 如果信息不足，宁可保留 `[待确认]`，也不要自行杜撰
- 如果用户要求“研发/测试可落地版本”，优先把状态、校验、异常、回流和数据项写完整
- 如果是模块型需求，避免把整页无关区域也扩写进去
