# 浏览器读取故障处理

用于蓝湖、Axure、墨刀、在线文档等可视化页面在浏览器工具中读取不稳定、DOM 和屏幕表现不一致、登录态混乱时的处理。

## 登录态复用

- 优先复用已有登录会话，不要一开始就要求用户重新登录
- 如果需要单独拉起可调试的 Chrome，会话目录要固定，这样下次能沿用 cookie 和登录态
- 推荐做法：使用固定 `user-data-dir` 启动 Chrome，再通过 CDP 或浏览器工具接管该会话
- 不要把账号密码明文写入 skill；优先复用浏览器会话或加密的登录状态

## 蓝湖 / Axure 页面读取规则

- `get text body`、`snapshot` 对这类页面通常只能稳定拿到外围 DOM、左侧目录、按钮、标注文本
- 画布中的正文、弹窗内容、局部交互状态，可能在 DOM 里缺失、滞后，或夹杂大量页面脚本
- 当页面等待约 10 秒后仍显示旧结果、旧页面、占位态、空白态或看起来没有更新时，先执行一次真实页面刷新，再重新取证；不要直接把刷新前的状态当结论
- 当 DOM 文本与屏幕显示不一致时：
  1. 以真实屏幕截图为主证据
  2. 以 DOM 文本作为目录、标题、页面树和补充信息
  3. 不要继续盲目重复读取同一份 `body` 文本

## 页面异常恢复顺序

遇到以下情况时，按顺序恢复：

- 蓝湖提示“由于打开过分享页，用户状态发生变化，为安全起见，请重新登录”
- 页面长时间停留在“正在加载”
- 浏览器工具拿到的还是旧页面状态
- 手工可见页面已经恢复，但工具读取仍是旧 DOM

恢复步骤：

1. 确认已接管的是正确的 Chrome 标签页
2. 将目标 Chrome 窗口置前
3. 对前台真实标签页执行浏览器级刷新，而不是只做页面级命令
4. 等待数秒后重新校验页面标题、目录和当前可见截图；如果 10 秒左右仍未更新，再做一次页面级状态核对，确认刷新结果是否真正生效
5. 如果 DOM 仍异常，但屏幕已恢复，切换为“截图裁图 + 局部取证”模式继续完成任务

不要在同一异常页面上无限刷新。浏览器级刷新完成后，如果仍然是旧 DOM 或骨架屏，直接进入截图模式继续完成需求整理。

## 截图取证规则

- 对 Axure/蓝湖画布页，优先截取：
  - 左侧页面树
  - 当前配置区或主内容区
  - 目标弹窗或目标区域
- 当整页截图中目标区域太小，看不清时，必须裁图放大再分析
- 如果右侧弹窗或底部按钮超出屏幕，先基于可见部分输出 `[明确]` 需求，再把缺失项列入 `[待确认]`
- 如果用户已经补充了清晰截图，优先把截图作为主证据，不要继续在 DOM 读取上消耗大量时间

## 错误总结

- 只依赖 DOM 文本，会误把脚本、骨架屏、旧状态当成当前页面内容
- 只做工具内刷新，不等于刷新了用户实际看到的前台标签页
- 对蓝湖/Axure 这类画布页，截图不是备用方案，而是正文读取的主方案之一
